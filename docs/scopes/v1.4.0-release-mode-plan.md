# Scope Plan: v1.4.0 Release Mode

- Status: Planned
- Date: 2026-03-02
- Target milestone tag: `v1.4.0`

## Objective

Add a production-oriented "release mode" preset that favors smooth rendering and lower terminal overhead while preserving visual quality.

## Why This Next

- The interactive TUI controls are stable now.
- Release mode should build on that stable baseline.
- It creates a clear profile for demos and daily usage, separate from learning/debug defaults.

## Why Not Other Next Scopes First

- Multi-backend capture support would increase complexity before performance baselines are locked.
- Recording/export features are less useful until runtime behavior is stable at target FPS.
- Advanced shading/filters can wait until release defaults are benchmarked.

## Proposed Scope (Step-by-Step)

1. Define release-mode behavior in config and docs.
   - What: add `--release-mode` flag and describe exact values it sets.
   - Why: explicit profile is easier to share and reproduce.
   - Why not implicit magic: hidden behavior is harder to debug and learn from.

2. Set practical default values for release mode.
   - What:
     - `render_fps=60`
     - `color_mode=ansi256`
     - `ramp=standard`
     - `show_metrics=false`
     - `max_cols=0`, `max_rows=0` (full terminal by default)
     - retain current camera `fps` unless user overrides
   - Why: balances smoothness and terminal throughput.
   - Why not truecolor by default: higher escape-sequence overhead on many terminals.

3. Add startup summary line for selected profile.
   - What: print one concise startup profile summary to stderr.
   - Why: helps verify active mode quickly.
   - Why not verbose logs: they interfere with clean TUI output.

4. Add benchmark script/checklist.
   - What: simple manual benchmark steps with expected FPS ranges.
   - Why: establishes baseline before further optimization.
   - Why not ad-hoc testing only: repeatability matters for tuning.

5. Validate cross-platform behavior.
   - What: smoke-test notes for macOS/Linux/Windows terminals.
   - Why: release mode should be reliable where this project targets usage.
   - Why not single-platform signoff: misses terminal capability differences.

## Acceptance Criteria

- `--release-mode` works with existing flags (existing explicit flags override preset values).
- Runtime controls still work identically in release mode.
- No regression in terminal cleanup and quit flow.
- `cargo test` passes.
- README documents release mode and tradeoffs.

## Deliverables

- Config flag and preset logic
- README update
- Scope completion note (`docs/scopes/v1.4.0-release-mode.md`)
- Version bump + tag (`v1.4.0`) when accepted
